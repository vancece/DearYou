# 给阿嬷一封来自云端的信

公众号文章，分为上、下两篇。

主线：介绍云开发（CloudBase）的 AI 能力生态 + CodeBuddy 原生集成云开发的开发体验。以「给阿嬷的情书」项目作为实际案例穿插其中。

### 一、开头：从电影说起

《给阿嬷的情书》票房近二十亿，豆瓣 9.3。强烈推荐还没看的朋友去看，小编二刷还是眼泪汪汪的。一封封侨批跨越山海，纸短情长。

电影里有个角色叫谢南枝，替人写了一辈子侨批。那些藏在心里、说不出口的话，经她的笔，变成信纸上一行一行的牵挂。会写信，是那个年代一门稀缺又珍贵的本事。

我就在想：今天我们大多识字，可那句"我想你"还是一样说不出口。如果每个人手边都有一个"谢南枝"，能把笨拙的心里话接成一封像样的家书——那会怎样？

于是有了这个小应用：「给阿嬷的情书」。你填几句大白话，AI 替你润色成一封有温度的侨批式家书。它不替你编故事，只把你说不利索的那点心意，写得让人鼻子一酸。

### 三、云开发：不只是"数据库加云函数"

很多开发者对云开发的印象还停留在"小程序后端"。

2026 年的云开发，已经是一个 AI 原生后端一体化平台：

- 云函数：Serverless 计算
- NoSQL 数据库：实时同步
- 文件存储 + CDN
- 静态网站托管
- AI 大模型：平台内置，不是第三方

关键区别：AI 模型是平台级能力，和数据库、云函数同一层级。不需要另外采购 API、不需要管 GPU、不需要管并发限流。

一个环境 ID，后端加 AI，全部搞定。

### 四、云开发的 AI 工具生态：三大协议，通吃所有 IDE

云开发支持三大主流 AI 协议：

| 协议 | 说明 |
|------|------|
| Chat Completions | OpenAI 兼容，最通用 |
| Responses API | OpenAI 新一代，原生状态管理 |
| Anthropic Messages API | Claude 原生协议 |

这意味着：所有支持自定义接口的 AI 编程工具，都能直接接入云开发的模型。

Cursor、CodeBuddy、Claude Code、Windsurf、Cline 这些开放自定义模型的工具，全部可以直接配；GitHub Copilot 则需企业版或特定设置才支持自定义端点。

配置只需要三个值：Base URL、API Key、Model Name。

贴配置代码示例。

更重要的是：你在 IDE 里写代码用的模型、你的应用里调用的模型，可以是同一个，走同一个计费池。

**应用接入云开发大模型**

着重讲清楚：代码里是怎么调用云开发 AI 的。

- 云函数中引入 `@cloudbase/ai` SDK
- 初始化时传入环境 ID，不需要额外的 API Key（云函数天然有环境鉴权）
- 调用 `ai.createModel()` 选择模型，然后 `generateText()` 或 `streamText()`
- 和调 OpenAI SDK 的体验几乎一样，但不需要自己管 Key、不需要担心额度耗尽后服务挂掉

贴一段云函数中调用大模型的核心代码。让读者看到：接入成本真的很低。

### 五、CodeBuddy 原生集成云开发

这是 CodeBuddy 和其他 AI IDE 最大的差异。

不只是"能用云开发的模型写代码"，而是：

**配置云开发模型，用云开发的额度进行开发**。在 CodeBuddy 设置里填入云开发的 Base URL 和 API Key，AI 编程助手就直接用云开发的模型来工作。你的云开发额度既用来驱动应用里的 AI 能力，也用来驱动你写代码时的 AI 助手——一份额度，开发和产品两头用。

**集成面板一键登录**。不需要手动查环境 ID、不需要配 .env、不需要去控制台复制粘贴。打开集成面板，登录，选环境，完事。

**在对话里直接操作云资源**。创建云函数、建数据库集合、配置安全规则、上传文件——不需要离开 IDE，不需要打开浏览器。

**一句话部署**。对 AI 说"帮我部署这个项目"，它自动检查环境、部署云函数、构建前端、上传静态托管、输出访问地址。

**AI 理解你的项目**。它会读 README、读项目结构，知道你有几个云函数、前端用什么构建工具、需要哪些环境变量。

开发、调试、部署、运维，全在一个窗口里完成。

贴 CodeBuddy 配置云开发模型的截图、集成面板截图、部署对话截图。

对比以前的流程：打开控制台 → 找到环境 ID → 复制 API Key → 手动写 .env → 手动 deploy → 切到浏览器看日志。现在这些全省了。

### 六、代码一瞥：如何教 AI 写侨批

简要展示 prompt 设计的核心思路。为了不让 AI 显得太"柴浪"（潮汕话，傻愣愣的意思），我们在 prompt 里下了不少功夫。

角色设定——"你是电影里那位替人代笔的谢南枝。你一辈子替人写侨批家书，深谙一个道理：最动人的信不靠华丽辞藻，而是把日常小事写得让人鼻子一酸。"

核心原则——用户写给你的每句话都是真实的回忆，是信的素材。你的工作是润色加工，不是凭空编造。

风格控制——三种基调各一句话概括：
- 温情：质朴克制，用细节代替抒情
- 幽默：晚辈跟长辈撒娇逗趣
- 正式：传统家书格式与措辞

贴一小段核心 prompt 代码。不过度展开。

一句话收尾：prompt 设计的好坏，直接决定输出质量。就像谢南枝的笔力，决定了一封侨批能不能让收信人落泪——同样几句平常话，写法不同，分量天差地别。

### 一、开头：会写不够，还得靠谱

让 AI 生成一段文字，如今谁都会了。调个接口，几行代码的事。

但做产品不一样。你需要它每次都返回固定格式，你需要它拒绝不合适的输入，你需要它在该克制的地方克制。

会写，和写得稳、写得靠谱，是两码事。这中间，就是"调 API"和"做产品"的距离。

本篇聊三件事：让 AI 输出可控、让 AI 守住底线、让 AI 成为你的开发搭档。

---

### 二、结构化输出：一次调用，多个结果

需求很具体：用户填写收件人称呼，但这个输入不可控——可能填正常的名字，也可能填不雅内容。

传统做法是前端加一层关键词过滤。但关键词库维护成本高、容易被绕过。

我们的做法：让 AI 在写信的同时，顺手审核收件人称呼。

具体方案：要求 AI 返回固定 JSON 结构——

```json
{"to": "审核后的称呼", "letter": "信件正文"}
```

在 prompt 中定义规则：正常称呼原样保留，不雅或违规的自动替换为基于关系的合理称谓。

云函数解析 AI 返回的 JSON，提取 to 和 letter。如果 AI 返回的不是合法 JSON（偶尔会发生），降级处理：整段当作信件正文，收件人用用户原始输入。

贴核心代码片段：JSON 解析 + 降级兜底逻辑。

关键经验：结构化输出让 AI 变得可编程。它不再只是一个"聊天对象"，而是一个可以嵌入业务流程的组件。

---

### 三、让 AI 当审核员

收件人审核只是一个切面。更大的话题是：AI 生成内容的安全边界。

我们的做法是在 prompt 层面做精准约束：

- 区分关系类型。不是一刀切"亲属之间不能写情话"，而是精确到：夫妻、爱人之间的表白正常润色；血亲关系出现恋爱内容才干预。
- 内容底线。信件不得包含违法违规、低俗、仇恨歧视的内容。

AI 语义审核和传统关键词过滤的对比：

| 维度 | 关键词黑名单 | AI 语义审核 |
|------|-------------|------------|
| 维护成本 | 高，需要不断补词库 | 低，prompt 定义规则 |
| 绕过难度 | 容易（谐音、变体） | 难（理解语义） |
| 误杀率 | 高 | 低（理解上下文） |
| 额外成本 | 独立调用 | 零（本来就要调 AI） |

关键洞察：如果你的业务本来就需要调用 AI 生成内容，让它顺手做审核，几乎不增加成本。审核逻辑写在 prompt 里，和生成逻辑一起执行。

经验总结：AI 的判断力需要人来校准。规则越精准，误杀越少，体验越好。过严和过松都是问题。

---

### 四、云开发 AI 模型：不只是文本生成

很多人对"AI 模型"的认知停留在"能聊天、能写文章"。

但在这个项目里，同一个模型做了三件不同的事：

- 文本生成：根据用户素材写出一封完整的家书
- 内容审核：判断收件人称呼是否合适，不合适就替换
- 格式控制：严格按 JSON 结构输出，让程序可以解析

此外，项目还预留了扩展接口（这一段可视篇幅一笔带过，不展开）：图片生成做个性化信纸背景、音乐生成给信件配乐、多语言翻译让侨批跨越语言边界——同一个 AI 基础设施，能力是顺着业务长出来的，不必每加一个功能就换一套服务。

云开发 AI 的定位不是"套壳某个大模型"。它是一个可编程的 AI 基础设施：多模型可选、控制台启用即用、统一计费、统一鉴权。

你不需要管 GPU 调度，不需要管并发队列，不需要管 token 限流。专注业务逻辑本身。

---

### 五、从"调 API"到"AI Agent"

AI 编程工具正在经历三个阶段：

第一阶段，代码补全。光标停在那里，AI 帮你续写下一行。Copilot 时代。

第二阶段，对话式编程。你描述需求，AI 生成一段代码，你复制粘贴。

第三阶段，Agent。AI 理解整个项目的结构，能操作外部资源，能自主执行多步任务。

CodeBuddy 加上云开发的原生集成，正在做第三阶段的事。

它不只是帮你写代码。它能创建云函数、配置数据库安全规则、构建前端、上传静态托管——这些操作全在对话里完成，不需要你切到终端或浏览器。

在这个项目里，我写了一份 deploy skill：告诉 AI 这个项目有哪些云函数、前端怎么构建、需要什么环境变量。之后每次部署，一句话就够了。

这改变的不只是效率。它改变的是开发者的角色——从"写每一行代码的人"，到"教 AI 做事、校验 AI 产出的人"。
