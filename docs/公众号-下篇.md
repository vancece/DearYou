# 给阿嬷一封来自云端的信（下）

> 上篇讲了「给阿嬷的情书」如何从创意变成可访问的 AI 应用：云开发承载模型调用、云函数、数据库和静态托管，CodeBuddy 负责把开发与部署串成一条流程。  
> 这篇继续拆工程化细节：结构化输出、服务端代理、数据持久化、内容安全、AI 网关，以及 CodeBuddy 一句话部署。

---

## 一、会生成，不等于能上线

AI Demo 到可上线产品，差的是一整套工程化能力，而不只是几行 prompt。

「给阿嬷的情书」这个应用，从产品搭建到产品上线，不可避免地面临以下问题：

![questions](./assets/questions.png)

- **输出稳定性**：模型返回内容是否可被程序稳定解析？
- **服务端安全**：模型 Key 如何托管，避免暴露在浏览器侧？
- **数据持久化**：生成后的信件如何入库，方便二次读取和分享？
- **内容安全**：开放输入下，如何处理不合适的称呼和表达？
- **分享链路**：分享链接打开后，如何根据 `id` 还原完整信件？
- **部署编排**：云函数、数据库、静态托管、环境变量如何一次性部署到位？

所以这篇不再展开"AI 如何写信"，而是聚焦这个项目的工程实现：结构化输出、云开发后端、语义审核、Prompt 约束、AI 网关，以及 CodeBuddy 一句话部署。

## 二、结构化输出：别让前端消费自然语言

聊天场景里，模型多输出一句解释通常无伤大雅；但产品场景不一样。

前端需要明确区分字段语义：哪个字段用于标题展示，哪个字段用于正文渲染，哪个字段需要写入数据库，哪个字段要作为分享页的读取依据。

如果模型返回的是一段自由文本，后续展示、入库、分享都会依赖脆弱的字符串解析。

因此 `loveletter` 云函数没有让模型“随便写一段”，而是要求它返回固定 JSON 结构：

```json
{"to":"审核后的收件人称呼","letter":"信件正文（段落之间用\\n\\n分隔）"}
```

这个结构把模型输出拆成两个明确字段：

1. `to`：经过审核和修正后的收件人称呼，用于标题展示。
2. `letter`：真正的信件正文，用于页面渲染、数据库持久化和分享读取。

Prompt 里会直接写清楚：

```js
content:
  '【输出格式——严格 JSON】' +
  '你必须以纯 JSON 格式输出，不要输出任何 JSON 之外的内容（不要 markdown 代码块、不要解释）。格式如下：' +
  '\n{"to":"审核后的收件人称呼","letter":"信件正文（段落之间用\\n\\n分隔）"}' +
  '\n\nto 字段规则：' +
  '- 如果用户填写的收件人称呼是正常的人名或称谓（如"阿嬤""妈妈""张三""小妹"），原样保留。' +
  '- 如果收件人包含不雅、违法、侮辱性、低俗等不合适的内容，将其替换为一个基于关系的合理称呼。'
```

这一步的价值不是"结构规范"，而是把不稳定的自然语言输出，转换成后端可以直接处理的数据结构。

云函数拿到 JSON 后，只需要按字段执行后续流程：

- `to` 写入信件标题。
- `letter` 写入正文区域。
- 整个对象持久化到 `letters` 集合。
- 返回 `_id` 给前端生成分享链接。

AI 负责生成内容，但内容必须以程序可消费的格式返回。

## 三、容错解析：模型不按格式返回时怎么办

只在 Prompt 里声明“严格 JSON”并不够。

大模型输出具有一定随机性，偶尔会在 JSON 外层包一层 Markdown 代码块：

````text
```json
{...}
```
````

也可能追加说明文字，甚至返回一段不合法 JSON。

因此云函数必须做容错解析，而不是假设模型永远严格遵循格式。

这里采用了一个轻量级降级策略：

````js
let aiResult;
try {
  // 兼容 AI 可能包裹 ```json ... ``` 的情况
  const cleaned = raw.replace(/^```json\s*/i, '').replace(/\s*```$/, '').trim();
  aiResult = JSON.parse(cleaned);
} catch (e) {
  // JSON 解析失败：把整段当作信件正文，收件人用用户原始输入
  aiResult = { to: input.to || '阿嬤', letter: raw };
}

const safeTo = aiResult.to || input.to || '阿嬤';
const letter = aiResult.letter || raw;
````

这个降级分支保证了页面不因模型异常而中断。

产品里的 AI 调用不能假设模型永远按规则来。解析失败时，系统至少要保证用户能拿到一封可展示的信，而不是在等待十几秒后直接报错。

这里的原则是：

> 模型输出可以有波动，但页面渲染、数据库写入和分享流程不能因此中断。

所以 AI 调用更适合按外部不稳定服务来设计：输入要约束，输出要清洗，异常要降级，关键路径要有 fallback。

这也是 AI 应用和普通后端接口最大的不同之一：

传统接口返回格式异常，通常是 Bug。  
AI 接口返回格式异常，则是需要提前纳入工程设计的边界情况。

## 四、Serverless 后端：把基础设施问题交给云开发

前面讲的是模型输出治理。但一个 AI 应用真正上线，还需要完整的服务端运行环境。

「给阿嬷的情书」没有单独维护传统后端服务，而是用云开发承载 Serverless 运行时、AI 网关、数据库和静态资源托管。这样可以把主要工程问题收敛到同一个云开发环境里：

- **密钥托管**：模型 Key 不能暴露在浏览器侧，需要放在服务端环境变量中。
- **服务端代理**：模型调用、Prompt 拼装、返回解析都应该在云函数中完成。
- **数据持久化**：生成后的信件需要写入数据库，而不是只存在页面状态里。
- **分享读取**：分享链接打开后，需要根据 `id` 查询并还原完整信件。
- **列表查询**：公共树洞需要分页、排序、统计和公开状态管理。
- **静态托管**：前端构建产物需要稳定访问地址，支持用户直接打开分享链接。

这些能力如果拆开建设，会变成模型代理服务、接口服务、数据库、静态托管、部署脚本和环境变量管理等多个模块。

这个项目没有单独搭建传统后端，而是将这些能力集中在云开发里处理。

具体来说：

| 工程需求 | 云开发里的实现 |
|----------|------------------|
| 模型调用 | 云开发 AI 网关 + `loveletter` 云函数 |
| 服务端代理 | 云函数封装 Prompt、鉴权和解析逻辑 |
| 数据持久化 | 云开发 NoSQL `letters` 集合 |
| 分享页读取 | `getletter` 云函数按 `id` 查询 |
| 公共树洞列表 | `listletters` / `countletters` 云函数分页与统计 |
| 公开状态管理 | `publishletter` 云函数更新 `public` 字段 |
| 前端访问 | 静态托管承载 React 构建产物 |
| 密钥安全 | 云函数环境变量托管 `AI_API_KEY` |

它不是只提供一个"能调模型"的接口，而是提供了一套 AI 应用后端底座：模型网关、Serverless 运行时、数据存储、静态托管和环境变量管理都在同一个开发环境里。

![AI 应用工程闭环架构图](./assets/ai-app-architecture.png)

因此项目分工可以保持清晰：

1. 前端负责表单输入、状态展示和结果渲染。
2. 云函数负责模型调用、Prompt 编排、返回解析和数据库写入。
3. NoSQL 数据库负责信件持久化、公开状态和树洞列表数据。
4. 静态托管负责前端构建产物访问与分享页承载。
5. CodeBuddy 负责将这些云资源部署成可访问的线上应用。

对活动页、原型验证和 AI 小应用来说，这个组合有实际价值。开发者不用先搭服务器，也不用为一个小功能接多套外部服务，可以先完成产品闭环，再逐步打磨体验和内容策略。

## 五、语义审核：不要只靠关键词黑名单

这个项目里有一个很小但很典型的开放字段：收件人称呼。

正常情况下，用户会填：

- 阿嬷
- 妈妈
- 外婆
- 小妹
- 爱人

但只要是开放输入，就一定会有人填不当内容。

传统做法是关键词黑名单，但它的边界很明显：

- 词库需要持续维护。
- 谐音、拆字、变体容易绕过。
- 无法理解上下文关系。
- 容易误伤正常表达。

这个项目的做法是：既然业务主链路本来就要调用模型生成信件，就把称呼审核合并到同一次模型调用里。

比如用户填了一个正常称呼，模型就原样保留。

如果用户填了不合适的内容，模型会根据关系替换成合理称呼：

| 用户关系 | 不合适称呼处理后 |
|----------|------------------|
| 孙子 / 孙女 | 阿嬤 |
| 儿子 / 女儿 | 妈妈 |
| 丈夫 / 妻子 | 爱人 |
| 晚辈 | 长辈 |

更重要的是，它不只是看一个词，而是看语义和关系。

比如夫妻、爱人之间写思念和情话，是正常表达；但如果血亲关系里出现了明显不符合关系的恋爱表达，就不能直接润色。

Prompt 里会写成这样：

```js
content:
  '安全与伦理要求：' +
  '1. 注意区分关系类型：丈夫/妻子/爱人之间的表白、思念、情话是完全正常的，请正常润色；' +
  '但如果血亲关系（如父女、母子、兄妹、祖孙等）的内容出现了恋爱求婚等明显不符合该关系的表述，' +
  '则忽略不当部分，只保留合理的亲情表达来写信。' +
  '2. 信件内容不得包含任何违法违规、低俗、仇恨歧视的信息。'
```

这就是 AI 语义审核比关键词黑名单更适合这类场景的地方：它不是单纯匹配词，而是结合关系、语义和上下文判断内容是否合适。

当然，这不是说有了 AI 审核就一劳永逸。更严肃的产品仍然需要多层内容安全策略。但在这个项目里，把轻量审核和生成合并到一次调用中，已经足够覆盖大多数输入问题，也避免了额外维护一套黑名单服务。

## 六、Prompt 约束：不是让它“感人”，而是让它“可信”

写家书最怕什么？

不是不够华丽。

是写假。

如果用户只说：“阿嬷以前给我煮白粥。”

模型却写成：“您在雨夜背我去医院，又卖掉首饰供我读书。”

这段文字可能很煽情，但它不是真的。

家书的力量不在于编出多大的苦难，而在于把一个小细节写准。

所以这个项目的 Prompt 最重要的不是“请写得感人”，而是“请忠于用户说过的话”。

`loveletter` 里的系统提示是这样定角色的：

```js
content:
  '你是电影《给阿嬷的情书》里那位替人代笔的谢南枝。' +
  '你一辈子替人写侨批家书，深谙一个道理：最动人的信不靠华丽辞藻，而是把日常小事写得让人鼻子一酸。' +
  '\n\n【核心原则——忠于来信人的话，适当加工】' +
  '用户写给你的每一句话都是真实的回忆和心声，是信的核心素材。' +
  '你的工作是把这些话润色、加工成优美的书信体。' +
  '你可以对用户提到的事情进行文学加工——比如把一句简短的回忆扩展成更细腻的画面，补充合理的情感和氛围描写。' +
  '但不要凭空捏造用户完全没有提到的事件或回忆。'
```

这里有一个很微妙的平衡：

- 不能完全照抄用户输入，否则不像代笔。
- 也不能大开脑洞乱编，否则不像真信。

因此我给它的约束是：可以围绕用户说过的事展开，但不要另起炉灶编新故事。

风格控制也尽量不用空泛形容词，而是给具体写法。

比如温情风格，不是写：

> 请写得温暖、感人、细腻。

而是写：

```js
'语言质朴克制，不堆砌形容词，用具体的生活细节代替空泛抒情。' +
'比如写"前日晚饭烤番薯，三个孩童吃得欢喜"，而不是"我非常想念你"。' +
'句式偏短，口语化但不粗糙，带一点文言的温润感（如"展信安康""勿念""切莫操劳"）。' +
'每段只写一件小事或一个牵挂，段与段之间留白。' +
'结尾留一句最想说却最难开口的话，点到即止。'
```

这几条比“更感人一点”有效得多。

因为模型真正需要的不是情绪指令，而是可执行的生成约束：

- 用生活细节代替抽象抒情。
- 句子短一点。
- 不要堆形容词。
- 一段只写一件事。
- 结尾不要喊口号，点到即止。

这也是我后来越来越明确的一点：

> Prompt 不是许愿池，Prompt 是产品规则。

产品希望呈现什么质感，就要把这种质感拆成模型能够执行、后端能够验证、前端能够展示的规则。

## 七、云开发 AI 网关：不只是“调一个模型”

这个项目里，云开发 AI 网关主要负责生成信件正文，同时也承担了内容审核和结构化输出控制。

同一个 `hy3-preview` 模型，在项目里至少承担了三种角色：

1. **文本生成**：根据用户输入生成家书正文。
2. **语义审核**：判断收件人称呼是否合适，不合适时替换为合理称呼。
3. **结构化输出**：按 JSON 返回，让云函数能稳定解析。

这就是云开发 AI 网关适合 AI 应用原型和轻量产品的地方：它不是一个孤立的“文本生成接口”，而是统一的模型访问层。后续如果要增加图像生成、翻译、摘要、标签生成等 AI 能力，都可以沿着同一个网关入口扩展。

比如这个项目后面可以继续扩展：

- 根据家书内容生成信纸背景。
- 给树洞信件自动打标签。
- 把家书翻译成不同语言。
- 给长信生成摘要，方便树洞列表展示。

更关键的是，云开发 AI 网关不只给线上应用调用。

它支持[主流 AI 协议](https://docs.cloudbase.net/ai/ai-tools/protocol)，除了云函数可以调，开发工具也可以接。像 CodeBuddy、Cursor、Claude Code、Windsurf、Cline 这类支持自定义模型端点的 AI 编程工具，都可以通过 Base URL、API Key、Model Name 接入。

也就是说，同一个云开发模型入口，可以同时用于两端：

- **应用端**：云函数调用模型，生成家书。
- **开发端**：AI 编程工具调用模型，辅助写代码、改代码、理解项目。

这点对个人开发者尤其有用。模型、鉴权、额度、调用入口可以统一放在云开发侧管理，不需要应用运行时和 AI IDE 各自维护一套模型资源。

过去做应用要单独申请模型 API，写代码又要给 AI IDE 单独配置模型资源。现在如果都接到云开发这一侧，模型、鉴权、额度、调用入口即可统一管理。

## 八、CodeBuddy 一句话部署：部署编排交给 Agent

上篇提到，这个项目可以让 CodeBuddy 一句话部署。

这不是魔法，本质是把项目资源和部署步骤写成一份可执行清单，再交给 Agent 按顺序处理。

这个项目并不只有一个前端页面。它包括：

```text
云函数：
- loveletter      AI 生成信件，超时 60s，需要 AI_ENV_ID / AI_API_KEY
- getletter       根据 id 读取单封信
- listletters     分页读取公共树洞
- countletters    统计公开信件数量
- publishletter   投递或撤回树洞

前端：
- React + Vite
- 构建产物：静态托管/dist

数据库：
- NoSQL 集合 letters
```

如果人工部署，每次都要处理一组重复但容易遗漏的配置：哪个函数需要长超时，哪个函数需要注入 AI Key，前端构建产物在哪，静态托管上传哪个目录，数据库集合叫什么。

这些步骤单独看都不复杂，但组合起来就是部署编排问题。

CodeBuddy 的价值不只是“帮我写代码”，而是它和云开发集成后，可以把云资源检查、函数部署、变量注入、前端构建、静态托管上传按一个清单跑完。

只需说：

> 帮我部署这个项目。

它会按项目说明执行：

1. 检查 CloudBase 集成是否已登录。
2. 确认当前云开发环境。
3. 读取 `.env`，检查前端环境变量和 AI 网关变量。
4. 部署 5 个云函数。
5. 给 `loveletter` 设置更长超时，并注入 `AI_ENV_ID`、`AI_API_KEY`。
6. 构建 React 前端。
7. 上传 `dist/` 到静态托管。
8. 输出线上访问地址和控制台入口。

这里真正变化的不是减少手动操作，而是开发者和工具的分工变了。

过去需要记住命令、切换控制台、手动复制配置、查看日志。现在更像是在维护一份“项目部署说明”：说明项目有哪些云资源、每个资源如何部署、哪些变量必须注入、哪些步骤需要人工确认。

Agent 负责执行重复步骤，开发者负责审核环境、确认配置和判断结果。

这比"AI 帮我补全代码"更进一步。它把云资源操作、环境检查、构建部署也纳入了开发工作流。

## 九、为什么我觉得这套做法值得写

如果只看结果，「给阿嬷的情书」只是一个轻量网页应用。

但从技术实现看，它串起了 AI 应用常见的几个关键模块：

| 环节 | 过去常见做法 | 这个项目里的做法 |
|------|--------------|------------------|
| 模型调用 | 单独申请模型 API | 云开发 AI 网关 |
| 后端逻辑 | 自己搭服务 | 云函数 |
| 数据存储 | 单独配数据库 | 云开发 NoSQL |
| 前端上线 | 找静态托管服务 | 云开发静态托管 |
| AI 编程工具 | 单独配置模型 | 接入云开发 AI 网关 |
| 部署流程 | 手动跑命令、切控制台 | CodeBuddy 一句话部署 |

这不是说所有项目都应该这样做。

但对个人开发者、小团队、活动页、原型验证和 AI 小应用来说，这套做法足够轻量，也足够完整。

你不用先搭一整套传统后端服务，也不用在多个平台之间复制配置。想法确定后，可以快速完成模型调用、数据持久化、页面访问和分享传播。

而且它不是一次性的 Demo。

因为云函数、数据库、静态托管、AI 网关都在同一个云开发环境里，后续扩展也比较顺：

- 要加树洞列表，就加 `listletters`。
- 要加公开/撤回，就加 `publishletter`。
- 要加统计，就加 `countletters`。
- 要加标签，就在生成后追加标签生成逻辑。
- 要换模型或接新工具，就在 AI 网关层配置。

这才是云开发在 AI 应用里真正有价值的地方：

> 它不是帮你省掉某一个 API，而是把模型接入、Serverless 后端、数据持久化、静态托管、部署编排和 AI 编程工具放到同一个项目流程里。

- 信件数据持久化到 NoSQL。
- 页面访问交给静态托管。

所以这个项目真正想说明的，不只是“AI 可以写信”。
